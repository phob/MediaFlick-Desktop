Switching the player backend between mpv and MPC-HC

A design for abstracting the external player behind one interface, adding an MPC-HC implementation driven over slave mode (WM_COPYDATA), and an honest account of which features survive the move and which do not.

TL;DR Architecture Control interface Feature matrix Hard blockers Implementation steps Risks & open questions

0TL;DR

MPC-HC is a viable secondary backend on Windows, driven via slave mode (/slave <hwnd> + WM_COPYDATA). Core playback — open, play/pause, stop, seek/resume, speed, end-of-stream → next episode — all map onto documented MpcApi.h commands.

The migration costs three features outright and degrades a handful more. The headline losses are chapter-marker injection for segments, external subtitle URLs, and the injected mark-watched hotkey — MPC-HC's API has no equivalent for any of them.

1Architecture: one interface, two implementations

Today the app talks to mpv directly through src/mpv/controller.rs and src/mpv/external.rs, with settings carrying a single mpvPath. The clean move is to extract the contract the rest of the app already relies on — load, control, observe — into a PlayerBackend trait, keep the existing mpv code as one implementor, and add an mpchc implementor beside it.

What the app needs from a backend

  • Commands in: load, pause, stop, seek, speed, volume/mute, audio/sub track, fullscreen, OSD text
  • Events out: file-loaded, position, duration, pause-state, end-of-stream, user-seek
  • Lifecycle: spawn idle, keep alive across episodes, restart on crash, quit

This is exactly the surface MpvController + ExternalMpv already expose internally — so the trait is a refactor, not a redesign.

Selection & settings

  • New setting playerBackend: "mpv" | "mpchc" (default mpv)
  • MPC-HC field: mpchcPath (slave mode is the only control path, so no method selector)
  • Settings dialog gains the backend switch (see the player-backend-settings.html sketch)
  • Factory at startup picks the implementor from the setting
// src/player/mod.rs — the shared contract
pub trait PlayerBackend {
    fn spawn_idle(&self) -> Result<()>;
    fn load(&self, req: LoadRequest) -> Result<()>;     // url + tracks + resume pos
    fn control(&self, cmd: PlayerControlCommand) -> Result<()>; // pause/seek/volume/…
    fn events(&self) -> Receiver<PlayerEvent>;            // position/eof/loaded/…
    fn capabilities(&self) -> Capabilities;               // what this backend supports
    fn shutdown(&self);
}

pub struct Capabilities {
    chapter_markers: bool,      // mpv: true   · mpc-hc: false
    external_subtitles: bool,   // mpv: true   · mpc-hc: false
    injected_hotkeys: bool,     // mpv: true   · mpc-hc: false
    absolute_volume: bool,       // mpv: true   · mpc-hc: false (relative only)
    pushes_position: bool,        // mpv: true   · mpc-hc: false (must poll)
}
Why a capabilities() probe matters: rather than scatter if backend == mpchc checks, callers ask the capability. The segment engine asks chapter_markers before drawing chapter pips; the progress reporter asks pushes_position to decide observe-vs-poll. This keeps the difference in one place and makes the UI able to grey out unsupported settings.

2How MediaFlick will talk to MPC-HC

The plan is slave mode only. MPC-HC also has an HTTP web interface, but it adds nothing we need and costs a second control path to maintain — see the contingency note below for the one scenario where it would re-enter the picture.

Slave mode The interface

  • Launch mpc-hc64.exe /slave <our_hwnd> once
  • MPC-HC replies CMD_CONNECT with its HWND
  • From then on we drive everything — including opening the next episode (CMD_OPENFILE) — through WM_COPYDATA. We never re-launch the exe to load a file.
  • MPC-HC pushes CMD_PLAYMODE, CMD_NOTIFYSEEK, CMD_NOTIFYENDOFSTREAM, CMD_NOWPLAYING back
  • Bidirectional — the closest analogue to mpv's IPC + observed properties

Needs a hidden message-pump window on our side to receive WM_COPYDATA (a small Win32 piece we don't have yet). The "launch once, then messages only" discipline is also exactly what sidesteps the #2895 double-load behaviour (see Risks).

Why not the HTTP web interface? Same capability ceiling (all three hard blockers below apply to it too), but strictly worse for us: poll-only with no pushed events, ambiguous end-of-file detection (which would undermine the next-episode handoff), a port to manage, and the user has to enable it manually in MPC-HC's options first. Its only edge — simpler code with no Win32 message pump — is undercut by the fact that we need a poll loop anyway (position isn't pushed even in slave mode). Building both means double the surface on an already-secondary backend.
HTTP contingency: keep it on the shelf for exactly one case — if slave mode turns out to be unworkable on the MPC-HC builds we target. That's a decision to make after testing slave mode, not work to do up front.

4The three hard blockers — and how to handle them

① Chapter markers for segments Not possible

mpv lets us rewrite chapter-list so intro/credits boundaries appear as pips on the seek bar. MPC-HC exposes no chapter-setting command and reads chapters only from the container.

Handling: degrade gracefully. Segment skipping still works (it's seek-based, not chapter-based), so users lose only the visual markers, not the behaviour. Gate the pips on capabilities().chapter_markers.

② External subtitle URLs Not possible

Jellyfin often serves subtitles as separate sidecar URLs; mpv loads them with sub-add. MPC-HC's API can only select tracks that already exist in the opened stream.

Handling: request a muxed/transcoded stream from Jellyfin with subtitles burned-in or embedded, so they arrive as in-container tracks selectable by CMD_SETSUBTITLETRACK. This changes the stream-URL negotiation for the MPC-HC path and may force transcoding for some subtitle formats. Flag external-only subtitle tracks as unavailable in the picker otherwise.

③ Mark-watched hotkey Not possible

The "w" key works because mpv runs an injected input section that fires a script-message back to us. MPC-HC has no way to register a host callback on a keypress.

Handling options:

  • Drop the in-player hotkey for MPC-HC; rely on natural EOF → next episode (which works via CMD_NOTIFYENDOFSTREAM).
  • Move the action into Jellyfin Web (a button / its own key handling in the CEF layer), independent of the player.
  • Global hotkey registered by MediaFlick at the OS level while MPC-HC is focused — heavier, and focus-fragile; likely not worth it.

5Implementation steps

  1. Extract the PlayerBackend trait

    Pull the command/event/lifecycle surface out of MpvController into src/player/mod.rs. mpv becomes src/player/mpv/… implementing the trait with zero behaviour change — pure refactor, existing tests stay green.

  2. Add the backend setting & factory

    Add playerBackend, mpchcPath, mpchcControl to settings.rs / client_settings.rs. A factory builds the right backend at startup. Wire the settings-dialog switch from the player-backend-settings.html sketch (Windows-only).

  3. Build the WM_COPYDATA transport

    Hidden message-pump window in src/player/mpchc/transport.rs: launch with /slave <hwnd>, handle CMD_CONNECT, encode/decode the pipe-delimited Unicode payloads, expose send + an inbound event channel.

  4. Implement the MPC-HC backend

    Map the trait onto CMD_*: open/play/pause/stop/seek/speed/tracks/fullscreen/OSD. Translate inbound CMD_PLAYMODE / CMD_NOTIFYSEEK / CMD_NOTIFYENDOFSTREAM / CMD_NOWPLAYING into PlayerEvents.

  5. Add the position-polling loop

    Since MPC-HC doesn't push time-pos, run a timer (~1 s) issuing CMD_GETCURRENTPOSITION and feed replies into the same progress pipeline the mpv path uses. Drive segment-skip checks off the same cadence.

  6. Wire capabilities() through the feature call-sites

    Chapter pips, external-sub picker entries, the mark-watched hotkey, absolute volume UI — each guarded by a capability flag and greyed out / hidden when the active backend can't do it.

  7. Stream-URL negotiation for MPC-HC

    Embed auth in the URL (api_key) instead of headers, and prefer a stream profile whose subtitles are embedded/burned so they're selectable in-player.

  8. Tests, changelog, docs

    Unit-test the WM_COPYDATA codec and the event translation; smoke-test a real MPC-HC. Add the [Unreleased] changelog entries (Added: backend switch; Changed: settings).

6Risks & open questions

Known risks

  • #2895 is a non-issue for us — the "spawns a new instance / loads twice" report (clsid2/mpc-hc#2895, closed) was API misuse: the reporter re-ran the exe to open files while also sending messages. The maintainer's guidance — "once you launched the exe as a slave you should control it with window messages only" — is precisely our design. Spawning a new instance on a second /slave launch is intentional. No version pinning needed on this account.
  • Auth in URL leaks the token into MPC-HC history / recent files.
  • Polling cadence trades fidelity for load — segment skips fire up to ~1 s late vs mpv's pushed updates.
  • Fork divergence — original mpc-hc vs the maintained clsid2 fork; target the latter and document the minimum version.

Open questions

  • Does CMD_SETPOSITION accept sub-second precision in the target build? (affects resume/skip exactness)
  • Is the MPC-HC build the user's install or one we bundle? (affects the "Get MPC-HC" affordance)
  • Acceptable to require transcoding when only external subtitles exist?
Recommendation: ship slave-mode-only behind a Windows-only switch, with chapter pips / external subs / mark-watched-hotkey cleanly degraded via capabilities(). Don't build the HTTP interface — hold it only as a contingency if slave mode proves unworkable on the target builds. Core viewing — open, resume, skip, auto-next — is fully intact; the losses are all secondary polish.

Grounded in src/mpv/controller.rs, src/mpv/external.rs, src/mpv/input.rs, src/jellyfin/playback_reporter.rs, src/jellyfin/media_segments.rs, and MPC-HC MpcApi.h (clsid2 fork).
